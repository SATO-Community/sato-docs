# ERC-20 Token Contract

## Overview

The SATO ERC-20 contract is responsible for token accounting.

It maintains balances, transfers, total supply, and protocol-authorized mint and burn operations.

The contract intentionally avoids discretionary administrative controls such as ownership, upgradeability, pause functions, blacklist functions, whitelist functions, or transfer taxes.

Monetary policy is not implemented directly inside the ERC-20 contract. Instead, monetary behavior is delegated to the protocol layer.

---

## Design Objectives

The ERC-20 contract is designed around the following objectives:

- Maintain ERC-20 compatibility
- Keep token accounting simple and transparent
- Separate token accounting from monetary policy
- Restrict minting and burning to protocol-authorized execution
- Avoid discretionary administrative controls
- Preserve predictable contract behavior after deployment

---

## Contract Architecture

The ERC-20 contract functions as the accounting layer of the SATO protocol.

It is responsible for:

- Account balances
- Transfers
- Allowances
- Total supply
- Mint recording
- Burn recording

It is not responsible for:

- Pricing
- Reserve accounting
- Bonding curve execution
- Governance
- Treasury management

This separation of responsibilities reduces complexity and makes the protocol easier to analyze.

---

## Protocol State Variables

The SATO ERC-20 contract intentionally maintains a minimal set of protocol-specific state variables.

These variables establish protocol identity, immutable deployment metadata, and narrowly scoped authorization while avoiding discretionary administrative control.

| Variable | Type | Description |
|----------|------|-------------|
| `minter` | `address` | Stores the protocol-authorized contract permitted to execute mint and burn operations. |
| `DEPLOYER` | `immutable address` | Records the deployment address authorized to initialize the protocol minter exactly once. |
| `GENESIS_BLOCK` | `immutable uint256` | Stores the Ethereum block number during deployment. |
| `GENESIS_HASH` | `immutable bytes32` | Stores the hash of the previous Ethereum block at deployment. |
| `RESTRICTIONS_FORBIDDEN` | `immutable bool` | Permanently indicates the intentional absence of blacklist, whitelist, pause, freeze, or transfer restriction mechanisms. |

The protocol intentionally minimizes mutable state.

Only the protocol authorization (`minter`) is initialized after deployment. All remaining immutable variables permanently preserve deployment information and cannot be modified.

---

## Authorization Model

Unlike conventional ERC-20 contracts, SATO does not implement an owner-controlled administrative model.

Instead, the contract authorizes a single protocol contract to perform mint and burn operations.

This authorization exists solely to allow deterministic protocol execution.

It does **not** provide discretionary monetary control.

Authorization is established exactly once during protocol initialization and is not intended to function as an ongoing governance mechanism.

This architecture separates token accounting from monetary policy while minimizing administrative trust assumptions.

## Function Reference

### constructor()

Initializes the ERC-20 token contract.

Responsibilities include:

- Setting the token name
- Setting the token symbol
- Recording deployment metadata
- Initializing immutable variables

The constructor executes only once during deployment.

---

### setMinter()

Assigns the protocol-authorized contract responsible for protocol execution.

Properties:

- Callable only by the deployer
- Callable only once
- Rejects the zero address
- Permanently establishes protocol authorization

This function exists exclusively for protocol initialization.

It is not designed as an administrative ownership mechanism.

---

### mint()

Creates new SATO tokens.

Access:

Protocol-authorized contract only.

Purpose:

The ERC-20 contract does not independently determine when tokens are created.

Instead, mint operations represent accounting updates resulting from deterministic protocol execution performed by the protocol layer.

---

### burn()

Destroys existing SATO tokens.

Access:

Protocol-authorized contract only.

Purpose:

Burn operations record supply reduction resulting from protocol-authorized redemption.

The function is not intended to provide discretionary balance modification capabilities.

## Events

The contract emits standard ERC-20 events for token transfers and approvals.

Additional protocol-specific events may be emitted during protocol initialization to improve transparency and auditability.

## Security Considerations

The ERC-20 contract intentionally omits many administrative mechanisms commonly found in modern token contracts.

These include:

- Ownership
- Upgradeability
- Proxy architecture
- Pause functionality
- Blacklists
- Whitelists
- Transfer taxes
- Administrative balance controls

The absence of these features reduces administrative complexity while improving predictability and transparency.

---

## Interaction with SatoHook

The ERC-20 contract is not responsible for monetary policy.

Protocol execution is delegated to the SatoHook contract.

SatoHook determines:

- Mint authorization
- Burn authorization
- Bonding curve execution
- Reserve accounting
- Issuance logic
- Redemption logic

The ERC-20 contract simply records the resulting state transitions.

Additional information is available in:

- `docs/protocol/04_SatoHook.md`

## ERC-20 Compliance

The contract follows the ERC-20 standard and therefore remains compatible with wallets, explorers, decentralized exchanges, analytics platforms, and other Ethereum infrastructure.

Protocol-specific authorization affects only minting and burning.

Standard ERC-20 transfer behavior remains unchanged.

---

## Design Decisions

Several architectural decisions intentionally distinguish SATO from conventional ERC-20 projects.

These include:

- Separation of accounting and monetary policy
- Immutable deployment metadata
- Minimal mutable state
- Protocol-authorized execution
- Elimination of discretionary administrative controls
- Transparent on-chain accounting

Together, these design decisions reduce trust assumptions while improving auditability.

---

## Summary

The SATO ERC-20 contract is intentionally designed to perform a single responsibility: token accounting.

Monetary policy, reserve accounting, and issuance mechanics are implemented by the protocol layer rather than inside the token contract itself.

This separation of concerns minimizes complexity, improves transparency, preserves ERC-20 compatibility, and reduces reliance on discretionary administrative control.
