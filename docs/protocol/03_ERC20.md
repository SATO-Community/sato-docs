# ERC-20 Token Contract

## Overview

The SATO ERC-20 contract implements the ERC-20 token standard and serves as the accounting layer of the SATO protocol.

Its responsibilities are intentionally limited to maintaining token balances, transfers, allowances, total supply, and protocol-authorized mint and burn operations.

Unlike many modern token contracts, the SATO ERC-20 contract intentionally excludes discretionary administrative features such as ownership, upgradeability, transfer restrictions, blacklist functionality, whitelist functionality, pause mechanisms, and transfer taxation.

Monetary policy, reserve accounting, and issuance logic are implemented by the protocol layer rather than the ERC-20 contract itself.

---

# Design Philosophy

The contract follows a minimalistic design philosophy.

Each component should perform one clearly defined responsibility.

The ERC-20 contract is responsible only for token accounting.

The protocol layer is responsible for monetary policy.

Separating these responsibilities improves readability, reduces complexity, and simplifies independent analysis.

---

# Design Objectives

The primary objectives of the ERC-20 contract are:

* Maintain full ERC-20 compatibility.
* Keep token accounting simple and deterministic.
* Separate accounting from monetary policy.
* Restrict minting and burning to protocol-authorized execution.
* Minimize administrative trust assumptions.
* Preserve predictable behavior after deployment.

---

# Contract Architecture

The ERC-20 contract functions as the protocol's accounting engine.

Its responsibilities include:

* Maintaining account balances.
* Recording token transfers.
* Managing allowances.
* Maintaining total supply.
* Recording protocol-authorized mint operations.
* Recording protocol-authorized burn operations.

The contract intentionally does **not** perform:

* Bonding curve calculations.
* Reserve accounting.
* ETH custody.
* Monetary policy decisions.
* Governance.
* Treasury management.

These responsibilities belong to the protocol layer.

---

# Protocol State Variables

The SATO ERC-20 contract introduces a small number of protocol-specific state variables.

These variables establish deployment metadata and protocol authorization while intentionally minimizing mutable state.

| Variable                 | Description                                                                                       |
| ------------------------ | ------------------------------------------------------------------------------------------------- |
| `minter`                 | Stores the protocol-authorized contract permitted to execute mint and burn operations.            |
| `DEPLOYER`               | Immutable deployment address authorized to initialize protocol execution.                         |
| `GENESIS_BLOCK`          | Immutable Ethereum block number recorded during deployment.                                       |
| `GENESIS_HASH`           | Immutable hash of the block immediately preceding deployment.                                     |
| `RESTRICTIONS_FORBIDDEN` | Immutable indicator documenting the intentional absence of administrative restriction mechanisms. |

Only the protocol authorization (`minter`) is initialized after deployment.

The remaining variables are immutable and permanently preserve deployment metadata.

---

# Authorization Model

The contract intentionally avoids an owner-based administrative model.

Instead, protocol execution is authorized through a dedicated protocol contract.

Authorization exists solely to enable deterministic protocol execution.

It does not grant discretionary monetary authority.

Protocol authorization is established during initialization and is intended to remain stable thereafter.

This model separates protocol execution from administrative ownership.

# Core Functions

## constructor()

Initializes the ERC-20 contract.

Responsibilities include:

* Initializing the ERC-20 token.
* Recording deployment metadata.
* Initializing immutable protocol variables.

The constructor executes exactly once during deployment.

---

## setMinter()

Assigns the protocol-authorized contract.

Properties:

* Callable only by the deployer.
* Callable only once.
* Rejects the zero address.
* Permanently establishes protocol authorization.

The function exists exclusively to initialize protocol execution.

It is not intended to serve as an ongoing administrative control mechanism.

---

## mint()

Records protocol-authorized token creation.

**Access**

Protocol-authorized contract only.

**Purpose**

The ERC-20 contract does not independently determine when new tokens are issued.

Instead, mint operations record issuance already authorized by deterministic protocol execution.

---

## burn()

Records protocol-authorized token destruction.

**Access**

Protocol-authorized contract only.

**Purpose**

Burn operations record supply reduction resulting from protocol-defined redemption.

The function is not intended to provide arbitrary administrative balance modification.

# Security Model

The ERC-20 contract intentionally minimizes discretionary administrative capabilities.

The implementation excludes features such as:

* Ownership
* Upgradeable contracts
* Proxy architecture
* Pause functionality
* Blacklists
* Whitelists
* Transfer taxes
* Administrative account restrictions

The resulting architecture reduces administrative complexity while improving transparency and predictability.

Protocol-authorized minting and burning should not be interpreted as discretionary owner privileges.

They represent deterministic protocol execution performed by the protocol layer.

---

# ERC-20 Compliance

The contract implements the ERC-20 interface and remains compatible with standard Ethereum infrastructure.

Compatible software includes:

* Wallets
* Block explorers
* Decentralized exchanges
* Analytics platforms
* Portfolio trackers
* Smart contract integrations

Protocol-specific authorization affects only minting and burning.

Standard ERC-20 transfer behavior remains unchanged.

# Relationship with SatoHook

The ERC-20 contract is intentionally separated from protocol execution.

The SatoHook contract is responsible for:

* Monetary policy
* Bonding curve execution
* Reserve accounting
* Issuance rules
* Redemption rules
* Protocol validation

The ERC-20 contract records the resulting token state transitions.

This separation of responsibilities simplifies protocol analysis and reduces implementation complexity.

# Design Decisions

Several architectural decisions distinguish the SATO ERC-20 implementation from conventional ERC-20 token contracts.

These include:

* Separation of accounting and monetary policy.
* Minimal mutable state.
* Immutable deployment metadata.
* One-time protocol authorization.
* No discretionary administrative ownership.
* Transparent on-chain execution.
* Protocol-based monetary authorization.

These decisions prioritize transparency, determinism, and reduced trust assumptions.

---

# Summary

The SATO ERC-20 contract is intentionally designed as a minimal accounting layer.

Its responsibility is to maintain token balances and record protocol-authorized state transitions while remaining fully compatible with the ERC-20 standard.

Monetary policy, reserve accounting, and issuance logic are delegated to the protocol layer, resulting in a modular architecture that emphasizes transparency, predictability, and deterministic execution.
